#### lib.dom
- 타입스크립트는 lib.dom 이라고 불리는 기본 정의 라이브러리를 로드해 전역 document 변수의 존재를 파악하고, 메서드를 사용할 수 있다.
- 이것은 자바스크립트 전역 객체를 설명하기 위한 목적을 가지고 여러 타입이 포함된 .d.ts 파일이라고 생각하면 된다.


#### 라이브러리와 타입 설정
```
// tsconfig.json
{
  "compilerOptions": {
    "types": ["vitest/globals"]
  },
  "include": ["src"]
}
```
- compilerOptions.types 를 사용함으로써 타입스크립트에게 추가적인 타입을 요청할 수 있다.
 - 위의 경우, 테스트 프레임워크에서 사용하는 메서드의 타입을 타입스크립트가 인식할 수 있도록 설정한다.

#### 타입스크립트의 유령적 정의 ( ghostly definition )
- 어떠한 객체가 브라우저의 전역 변수 혹은 타입스크립트의 기본 라이브러리가 아닐 경우에도 타입스크립트는 타입을 제안하고, 호출 시그니처를 보여주며, 오류를 표현하지도 않는다.
- 위의 경우, 컴파일은 되지만, 브라우저에서 작동하지는 않는다.
- include 옵션
	- include는 프로그램에 포함할 파일명 또는 패턴의 배열을 지정한다. - 타입스크립트 문서
	- include 옵션은 이 타입스크립트 구성을 적용할 모듈을 제어한다.
	- 만약, 타입스크립트 모듈이 include 옵션에 나열된 디렉토리 외부에 있는 경우, 해당 tsconfig.json은 해당 모듈에 전혀 영향을 미치지 않는다.
	- 마찬가지로 exclude 옵션을 사용하면 현재 구성의 영향을 받지 않아야 하는 파일 패턴을 필터링 할 수 있다.
	- include에 새로운 디렉토리를 추가하면, 포함한 모든 디렉토리에 영향을 미치도록 이 구성이 확장된다.
	- 위의 변경으로 compilerOptions에 추가한 프레임워크의 타입은 include에 포함한 디렉토리에 영향을 미치게 되지만, 이것은 모든 src 모듈에도 확장된다.
	- 이는 전체 소스 코드를 하나의 유령 저택으로 만들어, 수백 개의 유령 타입을 풀어놓은 것과 같다. 존재하지 않는 것들이 타입으로 정의될 것이고, 타입으로 정의된 것들은 다른 정의와 충돌할 수 있으며, 특히 애플리케이션이 시간이 자니고 성장함에 따라 타입스크립트 사용에 대한 전반적인 경험이 급감할 것이다.
	- **이를 해결하기 위해서는, 코드가 실행되도록 의도된 각 환경마다 tsconfig.ts를 생성해야 한다.**

- 모듈을 작성하는 목적은 중요하다.
	- 컴파일러는 스스로 추측할 수 없고, 사용자가 원하는 것을 컴파일러에게 알려야 한다.
	- tsconfig.ts를 통해 그 의도를 전달한다.
	- 타입스크립트는 중첩된 구성을 매우 효과적으로 처리할 수 있다.
	- 일반적인 옵션을 나열하는 루트 수준 구성입니다.
		다른 모든 구성은 여기에서 확장됩니다.
		"noEmit": true, "declarations": false를 설정하려고 합니다,
		그리고 "skipLibCheck": true를 설정하려고 합니다.
